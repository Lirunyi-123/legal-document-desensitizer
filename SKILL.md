# 法律文书脱敏工具 (Legal Document Desensitization)

## 概述

对法律文书中的敏感信息进行脱敏处理。采用 **规则引擎 + LLM 混合架构**：
- **规则层**：正则表达式快速识别身份证号、手机号、银行账号等结构化信息
- **LLM层**：AI识别规则覆盖不到的人名、公司名、地址、敏感案情等非结构化信息
- **替换策略**：语义替换（张三 → [当事人甲]），保留语义关系，律师可读

## 输入

接受以下格式的文档内容（粘贴或文件路径）：
- 合同/协议类文本
- 微信聊天记录、邮件等沟通记录
- 证据材料、谈话笔录等非结构化文本

**输入格式保留规则**：
- `.txt` 输入 → `.txt` 输出
- `.docx` 输入 → `.docx` 输出（保留段落结构）
- `.pdf` 输入 → `.txt` 输出（PDF格式保留较复杂，当前输出为纯文本）
- 未指定输出路径时，自动在原文件同目录生成 `原文件名_desensitized.扩展名`

使用示例：
```bash
# 自动保留格式（推荐）
python3 desensitize.py mask -f 合同.docx     # → 合同_desensitized.docx
python3 desensitize.py mask -f 判决书.docx   # → 判决书_desensitized.docx
python3 desensitize.py mask -f 聊天记录.txt  # → 聊天记录_desensitized.txt

# 指定输出文件
python3 desensitize.py mask -f 证据.pdf -o 脱敏后.txt
```

## 输出

1. **脱敏后文档**：所有敏感信息被替换为语义占位符
2. **脱敏映射表**：原始值 ↔ 替换值的对照表（用于内部还原）
3. **脱敏报告**：统计脱敏项数和类型

## 脱敏规则引擎（完整版）

### 第1层：正则规则（必须优先执行，执行前先扫描全文）

按以下顺序逐一扫描替换，**注意：每替换一个后要避免重复匹配**。

#### 身份信息类

| 类型 | 正则表达式 | 替换格式 | 备注 |
|------|-----------|---------|------|
| 身份证号 | `\d{17}[\dXx]` | `[身份证号]` | 18位，末位可能为X |
| 手机号 | `1[3-9]\d{9}` | `[手机号]` | 11位 |
| 固定电话 | `0\d{2,3}[-\s]?\d{7,8}` | `[固定电话]` | 含区号 |
| 出生日期 | `\d{4}年\d{1,2}月\d{1,2}日` | `[日期]` | 仅脱敏与个人关联的 |

#### 人名 / 公司名 / 地址类（规则层初步匹配）

| 类型 | 正则表达式 | 替换格式 | 说明 |
|------|-----------|---------|------|
| 人名 | 角色词（原告/被告/审判员/书记员等）+ 冒号/空格 + 2~3字姓名 | `[当事人甲]` / `[法官]` 等 | 仅匹配角色词后的人名，正文中的名字需LLM层处理 |
| 公司名 | `XXX有限公司` / `XXX公司`（含简称） | `[公司]` | 匹配完整公司名及3字以上简称 |
| 地址 | 省/市/区/路/号 地理层级结构 | `[地址]` | 匹配住所地、注册地址等 |

#### 联系信息类

| 类型 | 正则表达式 | 替换格式 |
|------|-----------|---------|
| 邮箱 | `[\w.+-]+@[\w-]+\.[\w.-]+` | `[邮箱]` |
| 微信号 | `微信号[：:]\s*\S+` 或 `微信[:：]\s*\S+` | `[微信号]` |
| QQ号 | `[Qq][Qq][:：]?\s*\d{5,12}` | `[QQ号]` |

#### 财务与机构代码类

| 类型 | 正则表达式 | 替换格式 |
|------|-----------|---------|
| 银行卡号 | `\d{16,19}` | `[银行账号]`（排除其他规则已匹配的） |
| 统一社会信用代码 | `[0-9A-Z]{18}`（匹配以9开头的18位） | `[统一社会信用代码]` |
| 组织机构代码 | `\w{8}-\w` | `[组织机构代码]` |

#### 案件程序信息类

| 类型 | 正则表达式 | 替换格式 |
|------|-----------|---------|
| 案号 | `\(?\d{4}\)?[﹒．\u4e00-\u9fa5]{1,10}\(?\d{1,6}\)?\d{0,3}号?` | `[案号]` |
| 执业证号 | `\d{17}` | `[律师执业证号]` |

### 第2层：LLM识别（规则层处理完后，对剩下的内容做语义识别）

> ⚠️ **注意：人名和日期已从规则层移除，完全由 LLM 层处理。**
> 规则层目前覆盖：身份证号、手机号、银行卡号、案号、统一信用代码、邮箱、微信号、QQ号、车牌号、公司名、地址、金额。
> 人名（所有自然人姓名）和日期（年月日）**不再由正则匹配**，交由 LLM 进行语义识别替换。

对规则层处理完的文本，用LLM识别以下**敏感信息**：

#### 需要识别的实体类型

1. **人员姓名**（规则层已移除，由LLM全权处理）
   - 自然人当事人的全名
   - 法定代表人、负责人姓名
   - 联系人姓名
   - 证人姓名
   - 历史文书中出现的所有人名（除公众人物的公开姓名外）

2. **日期**（规则层已移除，由LLM全权处理）
   - 出生日期、合同签署日期、判决日期等所有年月日
   - 替换为 `[日期]`，统一占位

3. **公司/机构名称**
   - 合同双方公司全名及简称
   - 第三方机构名称
   - 律所名称

3. **地址信息**
   - 精确到街道门牌号的地址
   - 户口所在地、现住址
   - 公司注册地址、通讯地址

4. **金额与财务敏感信息**
   - 大额合同金额（小额日常消费类不脱敏）
   - 付款账户信息
   - 薪资报酬

5. **案情中的敏感细节**
   - 不宜公开的私密事实
   - 商业秘密信息
   - 个人隐私细节

#### LLM识别规则

- **安全优先原则**：不确定是否敏感时，宁替勿漏
- **语义替换原则**：不用星号，用 `[角色/类型]` 格式
- **上下文感知**：根据文档角色给出有意义的占位符

### 第3层：语义替换规范

替换格式统一为 `[角色描述]`，示例如下：

```
原始: "张三于2024年1月向北京市朝阳区人民法院提起诉讼"
脱敏: "[当事人甲]于[日期]向[审理法院]提起诉讼"

原始: "甲方：北京华信科技有限公司，法定代表人：李四"
脱敏: "甲方：[合同甲方]，法定代表人：[法定代表人]"

原始: "我的微信号是 lawyer_wang，电话13800138000"
脱敏: "我的微信号是[微信号]，电话[手机号]"

原始: "尾款500万元应于验收后3日内支付至6222020200012345678"
脱敏: "尾款[金额]应于验收后3日内支付至[银行账号]"

原始: "原告金进跃，1985年8月15日出生"
脱敏: "原告[当事人甲]，[日期]出生"
```
注意：人名和日期由LLM层全权处理，规则层不干预。
```

### 第4层：生成脱敏映射表

输出格式为Markdown表格：

```markdown
# 脱敏映射表
| 序号 | 原始值 | 替换值 | 类型 | 出现次数 |
|------|--------|--------|------|---------|
| 1 | 张三 | [当事人甲] | 人名 | 15 |
| 2 | 北京华信科技有限公司 | [合同甲方] | 公司名 | 23 |
| 3 | 13800138000 | [手机号] | 手机号 | 2 |
| ... | ... | ... | ... | ... |
```

## 执行流程

### 步骤1：读取文档内容
- 如果输入是文件路径，读取文件内容
- 如果输入是文本内容，直接使用

### 步骤2：正则脱敏层
逐条执行所有正则规则，每次替换记录到映射表。

### 步骤3：LLM脱敏层
将步骤2处理后的文本交给LLM，要求识别剩余敏感信息并按语义替换。

**LLM提示词要点**：
```
你是一个法律文书脱敏专家。以下文本已经完成了结构化数据（身份证号、手机号等）的脱敏，
现在请你识别剩下的敏感信息并替换。

替换规则：
1. 人名 → [当事人甲]、[当事人乙]、[法官]、[律师]、[证人] 等，同一人用同一个占位符
2. 公司/机构名 → [合同甲方]、[合同乙方]、[第三方公司]、[审理法院] 等
3. 地址 → [地址]、[注册地址]、[联系地址] 等
4. 大额金额 → [合同金额]、[赔偿金额] 等
5. 其他敏感案情细节 → 概括性描述占位符

输出格式：
---
## 脱敏后内容
{脱敏后的完整文档}

## 补充映射表
| 原始值 | 替换值 | 类型 |
|--------|--------|------|
...
---
```
### 步骤4：合并映射表
将规则层的映射表和LLM层的映射表合并为完整映射表。

### 步骤5：输出保存
将脱敏后文档、完整映射表保存到 `archive/{时间戳}_脱敏结果/` 目录下。

## 脱敏质量标准

- ✅ 身份证号零遗漏
- ✅ 手机号零遗漏
- ✅ 所有自然人姓名替换
- ✅ 所有公司/机构名称替换
- ✅ 地址信息替换
- ✅ 映射表完整可追溯
- ⚠️ 语义占位符保留原角色的区分度（不同角色不同占位符）

## 安全注意事项

- 脱敏映射表是敏感文件，应妥善保管
- 上传AI时只上传脱敏后文档，**绝不**上传映射表
- 脱敏后的文档应做二次人工抽检
- 本技能不保存任何原始数据到外部

### ⚠️ 安全等级说明

本工具是分层脱敏系统，**不是一键魔法**。不同脱敏深度对应不同的安全等级：

| 安全等级 | 执行内容 | 替换了什么 | 还剩什么 | 能否上传给云端AI |
|---------|---------|-----------|---------|----------------|
| 🔴 不脱敏 | 什么都不做 | 无 | 全部敏感信息 | ❌ 绝不可上传 |
| 🟡 仅规则层 | 执行所有正则规则 | ✅ 身份证、手机、银行卡、案号等 | ⚠️ 人名、公司名、地址、金额、案情细节 | ⚠️ 有风险，不建议 |
| 🟢 规则层+LLM层(本地模型) | 规则层 + 调用本地Ollama等做语义脱敏 | ✅ 全部14类 | ✅ 全部替换 | ✅ 可以安全上传 |
| 🟢 规则层+LLM层(云端AI) | 规则层 + 交给云端AI做语义脱敏 | ✅ 全部14类 | ✅ 全部替换 | ✅ 可以，但云端AI那一步本身有数据暴露 |

**核心提醒**：
> 只跑规则层就把文件上传给AI——身份证号虽已替换，但人名、公司名、地址、金额、案情细节仍在泄露。
>
> **完整脱敏 = 规则层 + LLM层（本地或云端）**
